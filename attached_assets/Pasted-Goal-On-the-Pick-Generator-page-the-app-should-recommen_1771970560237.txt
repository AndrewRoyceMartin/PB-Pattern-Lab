Goal

On the Pick Generator page, the app should recommend a technique/strategy automatically based on the latest validation benchmark results.

That makes the generator smarter and keeps it aligned with evidence.

What the recommendation should do
If validation shows no predictive edge

Recommend:

Low Split-Risk / Anti-Popularity (practical mode), or

Random Baseline (if user wants pure chance)

If a strategy shows weak but unstable edge

Recommend:

Balanced mode (mix validated signal + anti-popularity)

Include a note like “weak/unstable; monitor over more windows”

If a strategy shows consistent positive edge

Recommend:

That strategy directly (e.g., most_drawn_last_50 or composite)

Show supporting stats from validation

Replit Instructions — Add “Recommended Technique” to Pick Generator
Step 1 — Add recommendation logic on the server
File to edit

server/analysis.ts (or create server/recommendation.ts if you want cleaner separation)

Add a function

Create something like:

recommendGeneratorTechnique(validationBenchmarkResult)

Inputs

Use the benchmark validation output you just added:

strategy stability summary

delta vs random by window

overall verdicts

optional diversity diagnostics

Output (recommended shape)

Return:

recommendedStrategy

recommendedMode (e.g. balanced, anti_popular, pattern_only)

confidence (low, medium, high)

reasonSummary (plain English)

evidence (small metrics object for UI)

Step 2 — Define recommendation rules (simple, strong rules first)
Suggested rule set (v1)
Rule A — No edge detected (most common)

If all predictive strategies are no_edge or underperforming:

Recommend:

mode = "anti_popular"

strategy = "random" OR "composite" with draw-fit weight near zero

Confidence: medium

Reason:

“No predictor beat random in walk-forward validation. Recommend low split-risk generation.”

Rule B — Weak edge only

If one strategy is weak_edge and others are no edge:

Recommend:

mode = "balanced"

strategy = best weak-edge strategy

Confidence: low

Reason:

“This strategy showed a small advantage in some windows, but not stable enough for pattern-only mode.”

Rule C — Possible edge

If one strategy is possible_edge across multiple windows:

Recommend:

mode = "pattern_only" or balanced

strategy = top performer

Confidence: medium (or high only if repeated strongly)

Reason:

“Consistently outperformed random across multiple windows.”

Rule D — Validation unavailable

If no benchmark exists yet:

Recommend:

mode = "balanced"

strategy = "random" or default baseline

Confidence: low

Reason:

“Run Validation Benchmark first to enable evidence-based recommendations.”

Step 3 — Persist or fetch latest validation benchmark result
Option A (fastest)

When user runs benchmark validation, store latest benchmark result in memory/server state and use it for recommendation.

Option B (better)

Store validation benchmark summary in DB (or a table/cache JSON) and fetch latest successful result.

Practical v1 recommendation

Use latest in-memory result first (fast to build), then persist later.

Step 4 — Expose recommendation via API
Add endpoint (recommended)

GET /api/generator/recommendation

OR extend:

POST /api/generate to return a recommendation block when requested

Endpoint response example (concept)

Return:

reason

benchmark evidence

fallback note if no validation run yet

This keeps frontend simple.
