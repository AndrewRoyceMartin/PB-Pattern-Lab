Step 2 — Implement frequency counting (training data only)
Important rule

When backtesting, never use future draws.

Inside the strategy function:

If current test draw index = i

Training draws = draws.slice(0, i)

For rolling variants:

last_50 => use trainDraws.slice(-50)

last_100 => use trainDraws.slice(-100)

Count separately

Main numbers frequency (1..35)

Powerball frequency (1..20)

Tie-breaking (important)

When two numbers have the same count, sort by:

higher count

more recent appearance (optional but useful)

lower number (stable fallback)

This keeps outputs deterministic.

Step 7 — Add rolling-window tests (important)

Don’t test only one backtest window.

In Validation settings UI, run at least:

20 draws

40 draws

60 draws

If your app supports one at a time, run them manually and compare.
If you want later, automate a “multi-window benchmark” mode.

Why

“Most drawn” can look good in one window by chance.

Step 8 — Add a plain-English summary for Most Drawn

In your summary/report text, add something like:

“Most Drawn (All-Time) is a descriptive strategy based on historical frequency and serves as a benchmark. It does not imply future draws are dependent on past draws.”

Then report whether it beat random out-of-sample.

This keeps the app credible.

Step 9 — Keep generator diversity healthy (important)

A common issue with frequency-based strategies is low diversity.

Add/verify these controls in generator config

top_n_games = 10

combo_pool_main_n (e.g. 14–18)

pb_candidates_n (e.g. 3–10)

If PB diversity is too low (e.g., always same PB), increase pb_candidates_n.
