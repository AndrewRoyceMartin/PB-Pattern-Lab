Task 1 — Replace fragile CSV parsing with a robust parser
Scope

Refactor upload/import parsing to use a proper CSV parser library instead of manual split(",").

Why

Manual splitting is fragile and can break on quoted fields, embedded commas, and format variations.

Implementation

Add a CSV parser package (e.g. csv-parse or papaparse)

Parse rows safely first

Keep your existing flexible column/header mapping logic after parsing

Preserve current import behavior for supported Powerball CSV layouts

Acceptance criteria

Existing CSV files still import correctly

Quoted fields and whitespace variations no longer break parsing

Import errors provide clear row-level messages where possible

Upload route code becomes simpler/safer

Task 2 — Fix benchmark averaging to use evaluated draw counts (rolling mode correctness)
Scope

Correct benchmark summary metrics so rolling walk-forward averages divide by the number of evaluated draws, not total test draws when some are skipped due to min training thresholds.

Why

Current averages can be understated if early draws are skipped in rolling mode.

Implementation

Track evaluatedDraws per strategy/window

Track skippedDraws per strategy/window (optional but useful)

Use evaluatedDraws as denominator for all averages and hit rates

Surface this metadata in benchmark tables

Acceptance criteria

Rolling benchmark summaries report correct averages

UI/API clearly show evaluated vs skipped draws

“Beats random” and deltas use corrected denominators

Fixed holdout mode remains unchanged

Task 3 — Improve permutation test methodology (v2)
Scope

Upgrade permutation testing to use a stronger null model and unbiased shuffling.

Why

The current v1 approach is useful but can be improved for statistical credibility.

Implementation

Replace sort(() => rng - 0.5) shuffling with Fisher–Yates

Permute outcomes across timestamps / draw order labels (or equivalent null-preserving scheme), not only within-draw order

Support selecting:

metric (best_total_score_delta, best_main_matches_delta, etc.)

scope (single_window, all_windows)

Clearly label method in response metadata

Acceptance criteria

Permutation results are reproducible with seed

API returns method metadata (shuffle type, scope, runs)

UI labels remain caveated (“supporting evidence only”)

No significant performance regression at default run counts

Task 4 — Persist benchmark runs and recommendation inputs in the database
Scope

Store benchmark results/configs so recommendations survive restarts and can be audited.

Why

In-memory latest benchmark state is fragile and disappears on restart/redeploy.

Implementation

Add a benchmark_runs table (JSON payload is fine for v1)

Store:

benchmark id

timestamp

config (mode, windows, minTrainDraws, seed, randomBaselineRuns)

summary results

optional detailed results JSON

status/success flag

Update recommendation engine to read latest successful benchmark from DB

Add stale-benchmark warning logic (e.g. older than X days)

Acceptance criteria

Recommendation still works after server restart

Benchmark history is queryable (at least latest + list endpoint optional)

Generator recommendation panel references persisted benchmark metadata

Missing benchmark state handled gracefully

Preserve endpoint paths and response shapes

Acceptance criteria

All existing endpoints still work

No route path regressions

Route modules are smaller and easier to navigate

Build passes without circular import issues

Why

Implementation

Acceptance criteria
