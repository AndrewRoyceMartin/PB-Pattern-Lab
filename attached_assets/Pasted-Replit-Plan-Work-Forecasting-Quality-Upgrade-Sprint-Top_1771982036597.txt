Task 1 — Add true rolling walk-forward benchmark mode
Scope

Upgrade benchmark validation so strategies can be tested in a real chronological replay mode (train → predict next draw → expand training → repeat), not just fixed holdout.

Requirements

Add benchmark mode selector:

fixed_holdout (existing behavior)

rolling_walk_forward (new)

In rolling mode:

test draw must be excluded from training stats/features

training set expands one draw at a time

Keep current scoring metrics unchanged so comparisons remain fair

Suggested API changes

Add to validation benchmark request:

benchmarkMode: "fixed_holdout" | "rolling_walk_forward"

Add to response metadata:

benchmarkMode

drawRangeUsed

minTrainDraws

Acceptance criteria

Rolling mode runs without leakage

Benchmark response clearly labels mode used

Results differ appropriately from fixed holdout when expected

Existing validation UI still works with fixed holdout mode

Task 4 — Add Bayesian-smoothed frequency strategies
Scope

Improve “most drawn” / “least drawn” style strategies by shrinking noisy observed frequencies toward the uniform baseline.

New strategies (minimum)

most_drawn_smoothed_last_50

most_drawn_smoothed_last_20

overdue_capped_smoothed (or recency_smoothed)

(optional) least_drawn_smoothed_last_50

Core idea

For each number:

observed frequency from window

prior belief = uniform probability

smoothed estimate = weighted combination of observed + prior

Use the smoothed score instead of raw count ranking.

Why this matters

reduces overreaction to short-term noise

improves stability in validation

gives a stronger benchmark than raw “hot/cold”

Acceptance criteria

Smoothed strategies appear in generator + validation lists

Benchmarks run successfully across windows

Strategy diagnostics show raw vs smoothed rank (or summary)

Tie-breaking remains deterministic

Task 5 — Add permutation significance testing layer (benchmark + Formula Lab support)
Scope

Add an experimental significance module to test whether observed benchmark/replay performance could plausibly arise by chance.

v1 target

Implement permutation test for one selected metric, e.g.:

mean_best_total_score_delta_vs_random
or

mean_best_main_matches_delta_vs_random

How it works (v1)

Preserve strategy generation pipeline

Break feature-outcome relationship (shuffle outcomes / test ordering as designed)

Recompute benchmark metric over many permutations

Compare observed result to null distribution

Return empirical percentile / p-value

API additions

permutationTest request block:

enabled

runs (e.g. 200)

metric

seed

Response:

observed metric

null mean/std

percentile

empirical p-value

caution text

Acceptance criteria

Permutation test can run on at least one strategy + metric

Results are reproducible with seed

UI displays significance result with caveat language

Recommendation engine ignores retrospective-only wins and does not treat untested significance as confirmed signal
